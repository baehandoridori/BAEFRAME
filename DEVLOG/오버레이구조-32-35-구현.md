## 계획서와 달랐던 지점 (필수 기록)

1. **[기준선 726 vs 687]** 계획서 헤더의 합계 "726 pass"는 개별 스크립트 수치(18+35+43+60+156+82+101+17+133+7+2+4+18+9+2 = **687**)와 불일치하는 산술 오류다. 실측 기준선은 687 pass/0 fail이며 개별 수치는 계획서와 문자 단위로 일치했다. 완료 보고·검증은 687 기준.

2. **[작업 1-(e) 파라미터명 epoch→owner]** 계획서 (e) 신규 테스트가 `function filterUnchangedMpvOverlayFields(state, epoch)`를 단언했으나, (a) 구현과 블로커 #1 확정(캐시 키는 epoch가 아닌 owner)에 따라 실제 파라미터는 `owner`다. 계획서 대로 `epoch`를 단언하면 신규 테스트 자체가 실패하므로 테스트를 `owner`로 정정했다.

3. **[작업 1-(f) hostSource 단언 배치]** `mpv-runtime-source.test.js`는 host 소스를 로드하지 않으므로, `playheadMirror.style.left = nextState.commentPlayheadLeft` 단언은 host 소스를 fs로 읽는 `mpv-overlay-host.test.js`에 배치했다. appSource 단언(`commentPlayheadLeft:`)만 mpv-runtime에 두었다.

4. **[작업 2-(b) 호스트 라우팅 인라인]** 계획서는 `sendDroppedPathToRenderer` 공용 헬퍼를 호스트 창 핸들러에서 재사용하도록 했으나, (c)의 호스트 테스트가 hostSource에 `open-from-protocol` 리터럴을 요구한다(공용 헬퍼 호출만으로는 리터럴이 host 소스에 없어 실패). 또한 `main/window.js`가 이미 두 호스트 모듈을 require한다. 따라서 확장자 라우팅 분기를 각 호스트 핸들러에 **인라인**하고(순환 require 회피 + 테스트 충족), `sendDroppedPathToRenderer` 헬퍼는 window.js에 유지해 메인 창 will-navigate에서 사용했다. 세 곳의 라우팅 로직은 동일 6줄 중복이나 허용 범위.

5. **[작업 4 loadVideoWithHtml5Fallback before 포맷]** 계획서 before 블록은 `loadVideo(filePath, { ...options, allowMpvPilot: false })`를 한 줄로 표기했으나 실제 코드는 멀티라인이었다. 의미는 동일하므로 스냅샷 변경 신호로 보지 않고 실제 멀티라인 포맷에 맞춰 편집했다.

6. **[작업 4-(b-5) showDriveVideoLoadingFeedback]** 계획서는 engineSwap에서 이 호출을 건너뛰라 했으나 `mpv-runtime-source.test.js:575`가 `const driveLoadingFeedbackShown = showDriveVideoLoadingFeedback(filePath, { preparedVideoPath });` 리터럴을 단언한다(인자·형태 고정). 호출을 제거·변경할 수 없어, 호출은 유지하고 engineSwap일 때 **즉시 hideVideoLoadingOverlay('drive')로 동기 억제**(같은 tick, 페인트 전 → 플래시 없음)했다.

7. **[작업 4-(b-2) 재들여쓰기]** eslint `indent: ["error", 2]`가 강제되어, 파괴 구간(~82줄)을 `if (!engineSwap) { ... }`로 감싸면서 `npx eslint --fix`로 블록을 재들여쓰기했다. app.js diff가 커졌으나(재들여쓰기 164줄 + 로직) 로직은 불변이다. 일반 로드는 `engineSwap` 기본 false라 모든 wrap이 항상 true → 동작 100% 불변. 파괴 구간 내 선언 변수(saved/proceed/shouldKeepMpvReviewFreeze)는 전부 블록 스코프라 밖으로 새지 않음을 확인했다. `hasExistingData`/`currentBframePath`만 호이스팅(let)했다.

8. **[작업 4 wrap 형태: 조건 병합 금지, 중첩]** `playlist-continuous-runtime.test.js:694`(`if (!preserveContinuousSession && continuousPlaybackState.active)`)·`:839`(`if (hasExistingData)`→`if (deferCollaborationStart)`)와 `lazy-bframe-creation-source.test.js`가 기존 if 리터럴을 단언하므로, 모든 engineSwap 스킵을 조건 병합이 아닌 **바깥 `if (!engineSwap)` 중첩**으로 구현했다.

9. **[작업 4 mpv-runtime 시그니처 단언 2건 갱신]** `loadVideoWithHtml5Fallback`에 `skipReviewTransition = false`를 추가해 `:1053`·`:1067`의 시그니처 정규식과 `:1055`의 `const reviewTransition = ...` 단언이 깨졌다. 의미 유지 방향으로 갱신(시그니처에 skipReviewTransition 반영, reviewTransition을 삼항 형태로).

10. **[작업 4 런타임 미검증 — 사용자 결정]** 이 실행 환경에서 mpv/Electron GUI 검증(계획서 rule 5, 검증 ①~⑩, 댓글 5개+ H.264 영상)을 수행할 수 없다. 계획서 rule 6("막히면 작업 4만 중단")에 따라 중단을 제안했으나, 사용자가 "작업 4도 구현, 검증은 사용자가"를 선택해 구현을 진행했다. **하이브리드 런타임 동작(엔진 전환·프레임 보존·undo/댓글/협업 유지·폴백)은 소스 단언 테스트로 검증되지 않으며, 배포 전 사용자 GUI 검증이 반드시 필요하다.** 미검증 시 설정 토글(hybridReviewEngine)로 끌 수 있고, 코덱 게이트·실패 폴백이 안전망이다.

## 코드 리뷰 결과 (병렬 서브에이전트 2인)

- **작업 4 loadVideo engineSwap (회귀 리뷰)**: 5개 속성(일반 로드 불변성·변수 스코프 호이스팅·브레이스/제어흐름·pause/resumeAutoSave 균형·engineSwap=true 경로) 전부 **실질 결함 없음**. 일반 로드는 wrap이 항상 true라 동작 불변, 호이스팅한 hasExistingData/currentBframePath만 함수 스코프, 나머지 wrap-local 변수 누출 없음.
- **작업 1 미러 diff**: 결함 없음. (복구 경로 `recoverMpvOverlayHostOnce`가 owner를 안 바꿔 diff 캐시를 무효화하진 않으나, 복구가 full unfiltered state를 force-push하므로 무해 — 사용자 가시 스테일 재현 불가.)
- **작업 2 호스트 경유 드롭 (인지된 스코프 제한)**: mpv 호스트 창을 통한 드롭은 `open-from-protocol`→`handleExternalFile`→`loadVideo` 경로라 `chooseDroppedVideoAction`(열기/합성 선택)·`addCompositionLayerFromPath`를 우회한다. **계획서가 명시적으로 이번 범위 밖(부록 C #2, "이번 범위는 열기 동작 복원")으로 둔 항목**이므로 의도된 동작. 영상 위 드롭 시 이미지=바로 열기(합성 레이어 선택지 없음), 다른 영상=바로 교체(선택 오버레이 없음). DOM 크롬 영역 드롭은 리치 경로로 정상. 후속(부록 C)에서 IPC에 드롭 좌표·선택지 전달로 통일 가능.
- **작업 3 마퀴 다중선택 abort no-op undo (인지된 minor 이슈, 미수정)**: `_onStrokeMarqueeSelect`의 `_saveToHistory()`가 제거·redraw·abort 검사보다 앞에 있어, abort 경로(async redraw 중 사용자가 그리기 시작 등 드문 타이밍)에서 no-op undo 엔트리가 1개 남아 Ctrl+Z를 한 번 더 눌러야 한다. 자기 주석("성공 확정 후에만 남긴다")과 어긋남. **미수정 사유**: `_saveToHistory`는 스냅샷 생성+push가 한 함수라, 스냅샷을 성공 후로 미루면 제거 후 상태를 캡처해 undo 시 오히려 획이 소실된다(데이터 손실). 제거 전 스냅샷은 성공 경로에 정확하며, abort no-op은 "마지막 스냅샷 pop" API 부재에서 오는 불가피한 minor 비용(형제 `_onStrokeSelectRequest`의 기존 동작과 동일). 위험한 undo 인프라 리팩터 없이 깔끔히 못 고쳐 계획서 코드 유지. 데이터 손실 없음, 드문 경로. 후속에서 취소 가능 스냅샷 도입 시 해소 권장.
