# 2026-02-10 로딩창 개발계획

## 문제 정의
- 앱 시작 직후 시각적 피드백이 없어 사용자가 실행 상태를 인지하기 어렵다.
- 특히 Slack 하이퍼링크(프로토콜/재생목록 링크)로 실행할 때 초기 응답이 없다고 느끼는 구간이 길다.

## 목표
- 앱 시작 직후 로딩창을 먼저 보여 실행 상태를 명확히 전달한다.
- 일반 실행과 딥링크 실행 모두에서 동일한 시작 피드백을 제공한다.
- 메인 화면이 준비되면 로딩창이 자연스럽게 닫히도록 전환한다.

## 설계 방향
- main 프로세스에서 `app.whenReady()` 이후 즉시 로딩창 생성.
- 시작 컨텍스트를 `app` / `deeplink`로 구분해 안내 문구를 다르게 표시.
- 딥링크 대상 파일이 있을 경우 파일명을 표시해 사용자가 진행 대상을 인지하도록 처리.
- 메인 윈도우 `ready-to-show` 또는 실패 핸들러에서 로딩창 정리.

## 구현 체크리스트
- [x] 일반 실행 시 로딩창 표시
- [x] 재생목록/프로토콜 실행 시에도 로딩창 표시
- [x] 런처 스타일 UI(로고, 상태 문구, 진행 애니메이션) 반영
- [x] 메인 화면 준비 후 로딩창 자동 종료
- [ ] 실제 배포 빌드에서 Slack 링크 실행 체감 시간 확인

## 검증 시나리오
1. 바탕화면/실행파일로 일반 실행 -> 로딩창 표시 확인
2. Slack 링크로 실행 -> 즉시 로딩창 표시 확인
3. 메인 화면 표시 시 로딩창 종료 확인
4. 파일 대상이 있는 경우 파일명 표시 확인

## 후속 개선 아이디어
- 로딩 단계(설정 로드/플레이리스트 파싱 등)를 퍼센트 기반으로 세분화
- 최근 열었던 프로젝트 정보 요약을 로딩창에 함께 표시
- 저사양 환경에서 애니메이션 강도를 자동으로 낮추는 옵션 제공
