## 배경 및 목적

Broadcast 기반 실시간 동기화 구현(Phase 1~5) 완료 후, 전반적인 코드 리뷰를 통해 10개의 잠재적 버그를 발견했다. 현재 상태에서 실사용 시 데이터 유실이나 동기화 실패가 발생할 수 있는 문제들이므로 일괄 수정한다.

## 버그 1: 네트워크 끊김 시 `_isConnected` 미갱신 (CRITICAL)

### 무엇이 문제인가

`liveblocks-manager.js:389-393`에서, 네트워크가 끊기면 `lost-connection` 이벤트의 `'lost'` 상태가 발생한다. 이때 `_connectionStatus`는 `'reconnecting'`으로 바뀌지만, **`_isConnected`는 `true`인 채로 유지된다.**

### 어떻게 동작하는가

```
정상 연결 중: _isConnected = true, _connectionStatus = 'connected'
     ↓
네트워크 끊김: _connectionStatus = 'reconnecting'  (변경됨)
               _isConnected = true                  (변경 안 됨!) ← 버그
     ↓
결과:
  - broadcastEvent() 호출 → room.broadcastEvent() 실행 → 전송 실패 (연결 없으니까)
  - 파일 감시 콜백 → isConnected가 true이므로 건너뜀 (app.js:7179)
  - 파일 감시도 안 되고 Broadcast도 안 되는 "양쪽 다 죽은" 상태
```

### 왜 오류가 나는가

`_isConnected = false`로 바꿔야 하는데 빠뜨렸다. `'failed'` 상태에서는 올바르게 `false`로 설정하고 있지만, `'lost'` 상태에서는 놓침.

### 수정

```javascript
// liveblocks-manager.js:390-392
if (event === 'lost') {
  this._connectionStatus = 'reconnecting';
  this._isConnected = false;    // ← 추가
  this._emit('connectionStatusChanged', { status: 'reconnecting' });
```

## 버그 2: 댓글 레이어 이벤트명 불일치 (CRITICAL)

### 무엇이 문제인가

`CommentSync`는 `layerAdded`와 `layerRemoved` 이벤트를 구독하고 있다 (comment-sync.js:54-55).
그런데 `CommentManager`는 이 이벤트를 **전혀 emit하지 않는다**. `layersChanged`만 emit한다 (comment-manager.js:831, 851).

### 어떻게 동작하는가

```
User A: 레이어 추가 버튼 클릭
  → CommentManager.addLayer() 실행
  → this._emit('layersChanged') 발생         ← 이 이벤트를...
  → CommentSync는 'layerAdded'를 기다리고 있음  ← 이 이름으로 듣고 있음
  → 이벤트 이름 불일치 → 리스너 호출 안 됨
  → Broadcast 전송 안 됨
  → User B는 새 레이어를 못 봄
```

### 왜 오류가 나는가

CommentSync를 작성할 때 `layerAdded`/`layerRemoved`라는 이벤트를 예상하고 리스너를 등록했지만, CommentManager에서 해당 이벤트를 emit하는 코드를 추가하지 않음.

### 수정

```javascript
// comment-manager.js:830-831 (addLayer)
this.layers.push(layer);
this._emit('layerAdded', { layer });      // ← 추가
this._emit('layersChanged');

// comment-manager.js:844-852 (deleteLayer)
this.layers.splice(index, 1);
if (this.activeLayerId === layerId) {
  this.activeLayerId = this.layers[0].id;
}
this._emit('layerRemoved', { layerId });  // ← 추가
this._emit('layersChanged');
this._emit('markersChanged');
```

## 버그 3: 답글 수정(replyUpdated) 동기화 누락 (CRITICAL)

### 무엇이 문제인가

댓글의 **답글 텍스트를 수정**하면, `CommentManager.updateReply()`가 `markerUpdated` 이벤트를 emit한다 (comment-manager.js:693). CommentSync의 `_onMarkerUpdated`는 이 이벤트를 받지만, **보내는 fields에 `replies` 배열이 포함되어 있지 않다** (comment-sync.js:113-122). 또한 `COMMENT_REPLY_UPDATED`라는 전용 이벤트 타입 자체가 없다.

### 어떻게 동작하는가

```
User A: 답글 텍스트 수정
  → CommentManager.updateReply() 실행
  → this._emit('markerUpdated', { marker }) 발생
  → CommentSync._onMarkerUpdated() 실행
  → COMMENT_MARKER_UPDATED 전송 (fields: text, x, y, colorKey...)
  → ❌ replies 필드가 없음 → 답글 수정 내용이 전달 안 됨
  → User B는 수정 전 답글을 계속 봄
```

### 수정

1. `CommentManager.updateReply()`에 `replyUpdated` 이벤트 추가
2. `CommentSync`에 `replyUpdated` 리스너와 `COMMENT_REPLY_UPDATED` 타입 처리 추가
3. Broadcast 수신 시 적용하는 `_applyRemoteReplyUpdate()` 메서드 추가

## 버그 4: 원격 키프레임 수신 시 캐시 미갱신 (CRITICAL)

### 무엇이 문제인가

`drawing-sync.js:302-304`에서 원격 키프레임을 받으면 `keyframe.canvasData`를 업데이트하지만, **`keyframe._cachedImage`와 `keyframe._cachedSrc`를 지우지 않는다.**

### 어떻게 동작하는가

```
로컬 저장 (drawing-manager.js:150-152):
  keyframe.canvasData = newData;
  keyframe._cachedImage = null;   ← 캐시 지움
  keyframe._cachedSrc = null;     ← 캐시 지움

원격 수신 (drawing-sync.js:303):
  keyframe.canvasData = newData;
  (캐시 안 지움)                   ← 버그!

재생 중 renderFrame():
  if (keyframe._cachedSrc === keyframe.canvasData) {
    // 캐시 히트 → 구버전 이미지 사용
  }
```

대부분의 경우 `_cachedSrc !== canvasData`라서 캐시 미스가 나고 새 이미지를 로드하지만, **재생 중 빠른 프레임 전환 시에는 캐시된 구버전 이미지가 표시될 수 있다.**

### 수정

```javascript
// drawing-sync.js:303 뒤에 추가
keyframe._cachedImage = null;
keyframe._cachedSrc = null;
```

## 버그 5: 대용량 canvasData(1MB 초과) 품질 축소 미구현 (MEDIUM)

### 무엇이 문제인가

`drawing-sync.js:187-191`에서 canvasData가 1MB를 초과하면 "품질 낮춰서 전송"이라는 경고 로그를 찍지만, **실제로는 품질을 낮추는 코드가 없다**. 원본을 그대로 전송하고, Liveblocks 서버가 거부하면 조용히 실패한다.

### 수정

1MB 초과 시 canvas를 낮은 품질로 재인코딩하는 `_reduceCanvasData()` 헬퍼 추가. 재인코딩 실패 시 원본 그대로 전송 (현재와 동일하게 동작).

---

## 버그 6: 레이어 생성 + 키프레임 도착 순서 문제 (MEDIUM)

### 무엇이 문제인가

User A가 새 레이어 생성 후 바로 그림을 그리면, 두 개의 Broadcast가 순서대로 전송된다:
1. `DRAWING_LAYER_CREATED` (레이어 생성)
2. `DRAWING_KEYFRAME_UPDATE` (그림 데이터)

그런데 네트워크 환경에 따라 **2번이 1번보다 먼저 도착할 수 있다.** 이 경우 키프레임을 적용할 레이어가 아직 없으므로 `return`으로 무시된다.

### 수정

`_applyRemoteKeyframe()`에서 레이어를 못 찾으면 200ms 대기 후 재시도:

```javascript
if (!layer) {
  await new Promise(r => setTimeout(r, 200));
  layer = retryLayers.find(l => l.id === layerId);
  if (!layer) return;
}
```

## 버그 7: Date 객체 직렬화 문제 (MEDIUM)

### 무엇이 문제인가

`comment-sync.js:307, 319-320`에서:

```javascript
createdAt: r.createdAt || new Date().toISOString()
```

`r.createdAt`가 Date **객체**이면 truthy이므로 `||` 뒤로 가지 않고, Date 객체가 그대로 Broadcast에 실린다. Liveblocks가 JSON 직렬화할 때 Date 객체는 `{}` 또는 `"2026-03-01T..."` 문자열로 변환될 수 있어 **일관성이 깨진다**.

### 수정

```javascript
createdAt: (r.createdAt instanceof Date ? r.createdAt.toISOString() : r.createdAt) || new Date().toISOString()
```

## 버그 8: 답글 이미지 첨부 직렬화 누락 (MEDIUM)

### 무엇이 문제인가

`_onReplyAdded()` (comment-sync.js:142-151)에서 답글을 직렬화할 때 `image`, `imageWidth`, `imageHeight` 필드가 빠져 있다. 이미지가 첨부된 답글을 보내면 상대방은 텍스트만 받는다.

### 수정

답글 직렬화에 이미지 필드 3개 추가. `_serializeMarker()`의 replies 직렬화에도 동일 적용.

## 버그 9: stop() 실패 시 리스너 누적 (LOW)

### 무엇이 문제인가

`app.js:3320-3327`에서:

```javascript
await liveblocksManager.stop();   // 이게 에러를 던지면
commentSync.stop();                // 이거 실행 안 됨
drawingSync.stop();                // 이거도 실행 안 됨
```

리스너가 정리되지 않아, 영상을 여러 번 전환하면 **이벤트 리스너가 누적**되어 한 번의 Broadcast가 여러 번 처리된다.

### 수정

`try/finally`로 감싸서 정리 보장.

---

## 버그 10: stop() 시 pending presence 미전송 (LOW)

### 무엇이 문제인가

`liveblocks-manager.js:213-217`에서 `stop()` 시 스로틀 타이머만 지우고, **미전송된 Presence 데이터를 flush하지 않는다.** Room을 나가기 전에 마지막 커서 위치나 상태가 전달되지 않아, 상대방 화면에 유령 커서가 남을 수 있다.

### 수정

타이머 정리 후 pending presence가 있으면 즉시 전송.
