## 2. 아키텍처 전제: mpv 3중 네이티브 창 구조 (25·27~31의 공통 배경)

**BAEFRAME** = 애니메이션 스튜디오용 비디오 리뷰/피드백 Electron 앱 (Electron 28 = Chromium 120, **mpv가 기본 재생 엔진**, HTML5 `<video>` 폴백, Canvas 2D 드로잉, Liveblocks 협업).
저장소: `C:\BAEframe\BAEFRAME`. 실행: `npm run dev`. 린트/포맷: `npm run lint:fix && npm run format` (2스페이스, 싱글쿼트, 세미콜론, `===`, 주석·토스트·커밋 메시지 한글).

mpv는 DOM이 아니라 **네이티브 자식 창**에 렌더링한다. z-order는 다음과 같고, 이것이 이번 피드백 대부분의 물리적 원인이다:

```
[아래] 메인 BrowserWindow — renderer DOM 전체 (컨트롤바, 타임라인, 드로잉 캔버스 "원본", 툴바)
  ↑
[중간] MPVEmbedHost (main/mpv-embed-host.js) — 투명 자식 BrowserWindow.
       창 핸들을 mpv에 --wid로 전달해 mpv가 여기에 영상을 그린다. --keepaspect-window=no (mpv-manager.js:103 부근)
  ↑
[위]   MPVOverlayHost (main/mpv-overlay-host.js) — 투명 자식 BrowserWindow.
       드로잉 합성 PNG·댓글 마커·토스트 등 DOM 요소를 "미러링"해 mpv 위에 보여준다. 항상 moveTop() 유지.
```

핵심 함의 (전부 코드로 확인됨):

1. **DOM은 mpv 위에 보일 수 없다.** 드로잉 캔버스·컨트롤바 등 메인 창 DOM은 mpv 창 뒤에 깔린다. 그래서 사용자에게 보여야 하는 오버레이는 MPVOverlayHost로 **미러링**한다(`getMpvOverlayState()`/`serializeMpvOverlayHtml()`, app.js 6454~6600). CLAUDE.md의 "어니언 스킨 BLOCKED: 캔버스 오버레이가 비디오 가림" 이슈가 이 제약의 기록이다.
2. **두 호스트 창 모두 클릭 관통이다**: `setIgnoreMouseEvents?.(true, { forward: true })` (mpv-embed-host.js:193, mpv-overlay-host.js:758). 마우스 입력은 메인 창 DOM에 도달하므로, mpv 뒤에 가려진 DOM(컨트롤바 등)도 클릭·호버 이벤트는 정상으로 받는다. 단 `forward: true` 때문에 mousemove가 호스트 창의 Chromium에도 전달되는데, 이것이 항목 27·29·31의 원인이다(작업 3).
3. **DOM 요소가 mpv 영역을 가리는 상황**(모달 등)에서는 mpv 창 자체를 숨긴다: `MPV_BLOCKING_OVERLAY_SELECTOR`(app.js:6003~6028, 24종 셀렉터 — `.drawing-tools.visible` 포함, `.controls-bar`는 **미포함**) → `hasBlockingOverlayForMpv()`(6121) → `syncMpvHostVisibilityWithDom()`(6137) → IPC `mpvSetHostVisible`. body 전역 MutationObserver(`installMpvBlockingOverlayObserver`, 6159~6174, attributeFilter: `['class','style','hidden']`)가 class/style 변화마다 이 동기화를 발화한다.
4. **가시성 판정은 computed style을 본다** — `isElementVisiblyBlockingMpv()`(app.js:6110~6119, 원문):
```js
  function isElementVisiblyBlockingMpv(element) {
    if (!element) return false;
    const style = window.getComputedStyle(element);
    if (style.display === 'none' || style.visibility === 'hidden' || Number(style.opacity || 1) === 0) {
      return false;
    }
    const rect = element.getBoundingClientRect();
    if (!doesRectOverlapMpvHost(rect)) return false;
    return rect.width > 0 && rect.height > 0;
  }
```
   → 어떤 요소든 `visibility: hidden`/`opacity: 0`이 되면 **셀렉터 목록을 바꾸지 않고도** 차단이 해제된다. 작업 4가 이 성질을 이용한다.
5. **드로잉 모드(B)는 "mpv 숨김 + 스크린샷" 모드다**: 드로잉 모드를 켜면 `.drawing-tools.visible`(도구 패널)이 3번 메커니즘으로 mpv 창을 숨기고, 대신 `showMpvDrawFreezeFrame()`(app.js:5264)이 mpv 스크린샷 PNG `<img class="mpv-draw-freeze-frame">`를 DOM에 깔아 정지 화면을 보여준다. 그 위에 드로잉 캔버스(DOM)가 있어 그리기가 가능하다. 이 구조가 항목 25의 배경이다(작업 4).
6. **HTML5 폴백 엔진에서는 25·27~31 증상이 없거나 다르다.** `isMpvPilotPlaybackActive()`(app.js:5239~5241) = `videoPlayer.engine !== 'html5' && document.body.classList.contains('mpv-pilot-mode')` 가드가 mpv 전용 로직을 나눈다. 모든 작업에서 HTML5 경로의 동작 불변을 유지할 것.

# 작업 1 — 항목 30: 드로잉 모드에서 마우스 휠 줌 허용

## 문제/요청 요약
드로잉 모드(B)에서는 마우스 휠로 화면 확대/축소가 되지 않는다. 사용자는 드로잉 모드에서도 휠 줌을 원한다(첨부 이미지: 줌 125% 상태로 드로잉 모드가 활성 — 즉 "줌 상태에서 드로잉"은 이미 가능하고, "드로잉 중 줌 변경"만 막혀 있다).

## 근거 (파일: `renderer/scripts/app.js`)
- 휠 리스너 원문 (5077~5085, 앵커: `// 비디오 영역 휠 줌`):
```js
  // 비디오 영역 휠 줌
  elements.viewerContainer?.addEventListener('wheel', (e) => {
    // 그리기 모드가 아닐 때만 줌 적용
    if (!state.isDrawMode) {
      e.preventDefault();
      const delta = e.deltaY > 0 ? -25 : 25;
      setVideoZoom(state.videoZoom + delta);
    }
  }, { passive: false });
```
- **근본 원인은 이 `!state.isDrawMode` 가드 하나뿐이다.** 전수 grep 결과 드로잉 캔버스에는 wheel 리스너가 없고(앱 내 wheel 리스너는 audio-waveform/timeline/composition 패널/위 리스너뿐), 드로잉 캔버스는 `#viewerContainer`의 자손이라 wheel 이벤트가 정상 버블링되어 위 리스너에 도달하지만 가드에서 무시된다.
- **줌 인프라는 드로잉 모드를 이미 지원한다** (수정 불요, 근거만 나열):
  - `setVideoZoom`(4907~4914) → `applyVideoZoom`(4936~) → `syncCanvasZoom`(4969~5005)이 드로잉 캔버스·레이어 캔버스·선택 오버레이·mpv freeze 이미지 전부에 동일 transform을 적용.
  - 드로잉 좌표 변환은 `getBoundingClientRect()` 기반(drawing-canvas.js `_getCanvasCoords`, 156~165)이라 CSS transform을 자동 흡수 — 현재도 "줌 → B 진입" 경로로 줌 상태 드로잉이 정상 동작한다.
  - 스페이스 홀드 패닝(피드백 14 구현)과의 결합: `canPanVideo()`(4920~4924)가 드로잉 모드에서 `state.isSpaceHeld && (videoZoom > 100 || !videoCenterLocked)`를 이미 허용.
  - mpv 엔진: 드로잉 모드에서는 freeze 이미지가 영상을 대신하며 `syncCanvasZoom` 대상(5000~5003)이고, `syncMpvVideoTransform`(6638~)이 네이티브 창에도 줌을 전파 — 별도 작업 불요.
- 스트로크 도중 줌 방지가 필요하다: 획을 긋는 중에 줌하면 `_getCanvasCoords`의 rect가 변해 진행 중인 선이 왜곡된다. 동일 목적의 선례가 Space 핸들러에 있음 — `app.js:10772` `if (drawingManager.drawingCanvas.isDrawing) return;`.

# 작업 2 — 항목 26: 단축키 `V`를 "select 도구 활성화"로 전환

## 문제/요청 요약
Animate처럼 드로잉 모드에서 `V`로 선택툴(마퀴 선택/이동/복사/붙여넣기 도구)을 활성화하고 싶다는 요청. 그런데 **`V`는 이미 "드로잉 모드 종료"로 하드코딩되어 있다**(도움말 오버레이에 "선택 모드 (그리기 종료)"로 문서화된 의도적 기능). 두 의미는 양립 불가하므로 이번 작업은 **기존 V 동작을 제거하고 요청대로 select 도구 활성화로 대체**한다.

**설계 결정(근거 포함):** 사용자의 기준은 Adobe Animate이고(Animate의 V = Selection tool), 피드백 문서 전반(13·16·23·26번)이 일관되게 Animate 동일화를 요구한다. 기존 "V=드로잉 종료"는 같은 사용자의 과거 요청 이력에 없는 개발 편의 기능이며, 드로잉 종료는 `B` 토글로 이미 가능하다. 따라서 대체가 요청 취지에 맞는다. 대체 사실은 커밋 메시지와 DEVLOG에 명시한다.

## 엣지 케이스 & 주의사항
- **"V로 드로잉 모드를 끄던" 기존 사용 습관이 사라진다** — 의도된 대체다(설계 결정 참조). 드로잉 종료는 `B`로 가능.
- **드로잉 모드 밖 V**: 이제 아무 동작 없음(기존: preventDefault만 하고 무동작). 실질 변화 없음.
- **활성 선택이 떠 있는 상태에서 V**: `selectDrawingTool` 첫 줄의 `commitActiveSelection()`이 먼저 확정 — 이미 select 도구인 상태에서 V를 다시 눌러도 확정만 일어나고 도구는 유지(안전).
- **커스터마이즈**: 레지스트리 등록으로 설정 모달에서 키 변경 가능 + `findShortcutConflict` 자동 충돌 검사에 포함된다.
- **`matchShortcut`은 `e.code` 기준**이라 한/영 입력 상태와 무관하게 동작한다. 기존 하드코딩과의 유일한 차이: 하드코딩은 `!e.metaKey`도 검사했지만 `matchShortcut`(user-settings.js:422~432)은 metaKey를 보지 않는다(Win+V도 매칭될 수 있음) — 앱의 다른 모든 레지스트리 단축키와 동일한 관례라 수용한다. 하드코딩 방식으로 되돌리지 말 것.
- 도구 버튼 title의 "V" 힌트 추가(index.html:598)는 하지 않는다 — 다른 도구 버튼 title 힌트(P/L/A/R/O)가 미구현 스테일 상태라, 일관성 작업은 범위 외(부록 C).

# 작업 3 — 항목 27·29·31: mpv 호스트 창 커서 경합 제거 (`forward` 삭제)

## 문제/요청 요약
세 증상 모두 "영상 영역 위에서 마우스를 움직일 때 커서가 고빈도로 다른 모양과 번갈아 바뀐다":
- **29**: 일반 화면(주로 줌>100% 상태)에서 hand(grab) ↔ 기본 화살표 반복.
- **27**: 드로잉 모드 select 도구로 이미지 선택 후 이동 시 "드르르륵" — 체감 최악 케이스.
- **31**: 드로잉 모드(crosshair)에서도 커서가 계속 튐.

## 근거 (전수 조사로 확정)

### DOM/CSS에는 진동 메커니즘이 없다
- **per-mousemove로 커서를 갱신하는 JS 코드는 앱 전체에 0건.** JS의 커서 설정은 전부 드래그 시작/종료 단발성(플레이헤드 스크럽 `ew-resize`, 마커 드래그 `grabbing` 등 — app.js/timeline.js/audio-waveform.js/split-view-manager.js). rAF/인터벌로 커서를 주기 리셋하는 코드도 없음(전 mousemove 핸들러 13곳 확인).
- 영상 영역 커서는 전부 **CSS 클래스 기반 정적 선언**이다: `.video-wrapper.zoomed { cursor: grab }`(main.css 6752~, `canPanVideo()` 참이면 `applyVideoZoom`이 `zoomed` 클래스 부여 — app.js 4953), `.drawing-overlay.active { cursor: crosshair }`(main.css 1109), `.video-wrapper.space-pan { cursor: grab }`(1125) 등. 클래스 토글은 줌/모드 변경 시에만 일어나며 mousemove마다 흔들릴 수 없는 연속 상태다.
- select 도구에는 커서 코드가 아예 없다(drawing-canvas.js의 select 처리 841~895와 drawing-manager.js 전체에 `cursor` 문자열 0건). 캔버스는 항상 CSS crosshair 하나다. hover(버튼 안 누른 이동)에서는 `_onPointerMove`(240~247)가 `_activeDrawPointerId !== e.pointerId` 가드로 즉시 리턴 — 아무 코드도 돌지 않는다.

### 진짜 per-mousemove 행위자: mpv 호스트 창 2개
§공통-2 구조에서 영상 위에는 투명 네이티브 창 2개(embed/overlay host)가 떠 있고, 둘 다:
```js
// main/mpv-embed-host.js:193 (원문. mpv-overlay-host.js:758도 동일)
    this.window.setIgnoreMouseEvents?.(true, { forward: true });
```
Electron에서 `forward: true`는 클릭은 통과시키되 **mousemove를 그 창의 Chromium 페이지에도 계속 전달**한다(Windows 전용 옵션). 두 호스트의 페이지는 커서 스타일이 없는 빈/미러 문서(embed: 빈 body, overlay: `pointer-events:none` body — mpv-overlay-host.js 47~56)라 계산 커서가 **기본 화살표**다. 그 결과 마우스가 영상 위에서 움직일 때마다:
1. 메인 창 Chromium이 DOM 커서를 OS에 설정 — zoomed면 **grab**, 드로잉 모드면 **crosshair**
2. 호스트 창 Chromium이 전달받은 move를 처리하며 **화살표**를 설정

OS 커서는 전역 하나이므로 두(세) 프로세스가 번갈아 SetCursor → **깜빡임**. 세 증상의 발생 위치(영상 위에서만)·시점(mpv 기본 엔진화 07-09/10 직후 07-13 일괄 제보)·양상(이동 중 고빈도 전환)이 전부 이 메커니즘과 부합한다. mpv 자체 커서 개입은 `--cursor-autohide=always`(mpv-manager.js:103 부근)로 이미 봉인되어 있다.
- 항목별 가중: **29** = grab과 화살표의 대비가 커서 가장 눈에 띔(첨부 GIF도 줌 상태). **27** = 선택 드래그 중 48ms마다 드로잉 합성 PNG 인코딩+IPC(`scheduleMpvOverlayStateSync({liveDrawing:true})` 경로)가 메인 렌더러를 스톨시켜 호스트 창의 화살표가 이기는 시간이 길어짐(이 가중 인과는 추론 — 검증 절차로 확인). **31** = 드로잉 모드에선 호스트 창이 숨겨져 있는데도 증상이 보고됨 — 숨긴 창의 forward 동작이 변수(부록 A-1에서 판별).
- `forward`를 제거해도 잃는 것이 없다: **두 호스트 페이지 모두 마우스 이벤트 리스너가 0건**(전수 grep 확인)이라 forward가 주는 기능 이득이 없고, 클릭 관통(`setIgnoreMouseEvents(true)`)은 유지되므로 하단 호버 감지·드로잉·마커 클릭 등 메인 창 DOM 입력은 그대로다.

## 근본 원인

## 엣지 케이스 & 주의사항
- **클릭 관통 자체는 불변**: `setIgnoreMouseEvents(true)`가 유지되므로 영상 위 클릭/드래그/호버는 계속 메인 창 DOM으로 간다. 전체화면 하단 80px 호버 감지(app.js 7827~)·비디오 패닝·코멘트 마커 클릭·드로잉 입력 전부 이 경로다 — 검증 절차에 포함.
- **Windows 전용 옵션**: `forward`는 Windows에서만 의미가 있고 이 앱은 Windows 전용 배포다. 제거는 타 플랫폼에 무영향.
- **오버레이 미러 상호작용 없음**: 오버레이 호스트 페이지는 원래 `pointer-events:none`이라 forward 유무와 무관하게 상호작용이 없었다.
- **이 수정은 27의 "안정화" 요청 중 커서 증상만 다룬다.** select 도구에 hover 커서(선택 영역 위 `move` 등)를 새로 넣는 것은 새 깜빡임 위험이 있어 범위 외(부록 C). 48ms PNG 인코딩 부하 최적화도 범위 외(부록 C) — 단 커서 경합이 사라지면 스톨은 "커서 깜빡임"이 아니라 최악의 경우 미세한 드래그 버벅임으로만 남는다.
- **31의 잔존 가능성**: 드로잉 모드에서는 호스트 창이 숨김 상태다. 숨긴 창이 forward를 계속 수행했는지(H1) 특정 시퀀스에서 호스트가 안 숨은 채 남았는지(H2)는 이 수정 후 관찰로 판별한다 — 어느 쪽이든 forward 제거가 커서 경합을 없앤다. 잔존 시 부록 A-1의 데이터 수집 지침을 따르고 **임의 수정을 확대하지 말 것**.

# 작업 4 — 항목 25: 드로잉 모드에서 재생 시 영상이 보이게

## 문제/요청 요약
드로잉 모드(B) 상태에서 재생하면 **소리만 나오고 화면은 정지 프레임에 머문다**. 드로잉 모드를 끄면 정상. 사용자는 "드로잉 모드를 활성화한 상태에서의 영상 재생"이 정상 동작하길 원한다 — 이 사용자는 피드백 14(이미 구현됨)로 "드로잉 모드 안에서 스페이스 탭 = 재생 토글" 워크플로를 확보한 상태라, 재생 시 모드를 강제 종료하는 우회는 요청 취지에 어긋난다.

**설계 결정(근거 포함):** 드로잉 모드를 유지한 채 **재생 중에만 freeze 스크린샷과 도구 패널을 잠시 감춰 mpv 영상이 보이게 하고, 정지하면 원상 복귀**한다. 검토한 대안: ① 재생 시 드로잉 모드 자동 종료 — 구현 최소지만 피드백 14 워크플로(모드 유지 재생/정지 반복) 훼손으로 기각. ② 재생 중 스크린샷 주기 갱신 — 스크린샷 IPC가 건당 파일 IO+base64라 초당 수 회로도 슬라이드쇼 수준이라 기각. ③ 차단 셀렉터/판정 로직에 재생 예외 추가 — 재발 이력 5회의 민감 영역 직접 수정이라 기각(§공통-6-2). 채택안은 §공통-2-4의 computed-style 판정 성질을 이용해 **셀렉터·observer·판정 로직을 전혀 건드리지 않는다**. 그려둔 획은 재생 중에도 기존 오버레이 미러(드로잉 합성 PNG → MPVOverlayHost)로 영상 위에 계속 표시된다.

> 타이밍 설계: (a)의 `transition: none` 덕에 클래스 추가 즉시 computed 상태가 `visibility:hidden`이 되어, observer의 rAF 판정이 **곧바로** 차단 해제 → 호스트 표시 IPC를 낸다. 그 IPC~창 표시까지 수십 ms 동안은 freeze가 화면을 지키고(정지 프레임 = 직전 화면이라 시각 연속), mpv 창이 뜨면 freeze는 그 뒤에 가려진다. `MPV_OVERLAY_FADE_OUT_SYNC_DELAY_MS`(= 350, app.js:82의 기존 상수) 후 상태 정리를 위해 제거한다. `scheduleMpvOverlayStateSync({ force: true })`는 호스트가 다시 보일 때 그려둔 획 미러가 최신 상태로 얹히도록 즉시 1회 동기화한다.

## 엣지 케이스 & 주의사항
- **`applyDrawModeState`의 기존 라인은 삭제·이동하지 않는다** — mpv-recovery-source.test.js:63~75가 함수 본문을 정규식으로 고정하고 있다(진입 시 pause+freeze, 해제 시 remove). (e)는 라인 추가만 한다.
- **frameUpdate 디바운스와의 공존**: 재생 중 `scheduleMpvDrawFreezeRefresh`는 여전히 매번 리셋되어 미발화한다. freeze가 (b)의 지연 제거 전까지 mpv 뒤에 남아 있어도 시각 영향이 없다(가려짐). 정지 시에는 마지막 스케줄이 160ms 후 발화해 (c)의 즉시 스크린샷과 겹칠 수 있으나 토큰이 마지막 것만 반영한다.
- **재생 중 드로잉 입력**: mpv 창이 위에 있으므로 캔버스 스트로크는 화면에서 직접 보이지 않고 48ms 미러로만 보인다. 재생 중 그리기는 원래 주 워크플로가 아니고(정지 후 그리기), HTML5 경로도 동일 시맨틱이므로 추가 처리하지 않는다 — 동작 확인만 하고 개선은 범위 외.
- **연타 경합(재생↔정지 빠른 반복)**: 양방향 모두 최종 상태 기준으로 수렴한다. ① play 쪽: freeze 지연 제거가 `videoPlayer.isPlaying` 재확인 후에만 실행 — 350ms 안에 정지로 바뀌면 pause 경로가 만든/유지한 freeze를 지우지 않는다. ② pause 쪽: `.finally`의 `videoPlayer.isPlaying` 가드 덕에, 스크린샷 IPC가 도는 사이 재생이 재개됐으면 패널 복원을 건너뛴다(이 가드가 핵심 — 없으면 "재생 중 + 호스트 숨김 + freeze 없음 = 검은 화면" 상태가 만들어진다). 최종 pause면 마지막 스크린샷이 토큰으로 반영되고 `forceMpvHostVisibilitySync`가 숨김을 확정한다. observer 동기화는 `mpvHostLastRequestedVisible` 캐시로 중복 IPC를 걸러낸다(6144~6147).
- **정지 → 호스트 숨김 사이의 짧은 전이**: (c)에서 클래스 제거 후 호스트가 실제로 숨기까지(트랜지션 0.2s + 강제 재평가) 사용자는 mpv의 정지 프레임을 본다 — freeze와 같은 프레임이라 시각적 단절은 없고, 패널이 페이드인으로 돌아오는 것만 보인다. 즉 "정지 직후 패널이 살짝 늦게(≤0.35s) 나타나는" 것은 이 설계의 의도된 특성이다.
- **다른 차단 요소와의 중첩**: 재생 중 모달 등 다른 차단 셀렉터 요소가 열리면 기존 메커니즘대로 호스트가 숨는다 — 본 수정과 독립적으로 기존 동작 유지.
- **HTML5 엔진 회귀 없음**: 모든 추가 블록이 `isMpvPilotPlaybackActive()` 가드 안이다. HTML5 경로는 원래 드로잉 모드 재생이 정상( `<video>`가 DOM에 그려지고 캔버스가 위에 겹침)이며 불변.
- **협업 원격 재생**: 원격 참가자가 재생을 트리거해도 로컬 `videoPlayer`의 play/pause 이벤트로 동일 경로를 탄다. 추가 처리 불요.
- **어니언 스킨 BLOCKED**(CLAUDE.md)와 무관 — 어니언 스킨 캔버스는 건드리지 않는다.

# 작업 5 — 항목 28: 전체화면 컨트롤바 — 검은 여백 게이트 + 오버레이 미러링 + 스냅 인셋

## 문제/요청 요약
전체화면(F)에서 화면 하단에 마우스를 대면 컨트롤바(재생 컨트롤+시크바 — 사용자는 "아래 타임라인"이라 부름)가 올라오면서 영상 영역이 줄어든다. 사용자 불만 3가지:
- **(a)** 영상 위아래에 검은 여백(레터박스)이 충분한데도 화면 전체가 줄어든다 — **여백이 있으면 그 위에 바로 컨트롤바가 뜨길 원함** (핵심 요청, 첨부 이미지도 이 케이스).
- **(b)** 줄어드는 애니메이션이 "찌글찌글 비율이 깨지듯" 부자연스럽다.
- **(c)** 애니메이션 중 드로잉과 영상이 어긋나 보인다.

**설계 결정(근거 포함):**
1. **(a) 여백 게이트**: 영상 하단 검은 여백이 컨트롤바를 다 수용하면 인셋을 0으로 — 화면·영상·드로잉이 전혀 안 움직이고, 컨트롤바는 여백 위에 표시된다. 이 경로에서는 (b)(c)도 원천 소멸.
2. **(a) 표시 방법 — 미러링**: mpv 창이 DOM 위이므로(§공통-2-1) 게이트 시 컨트롤바 DOM은 mpv에 가려진다. DOM을 직접 위에 올리는 것은 구조적으로 불가능하므로 **기존 MPVOverlayHost 미러링 인프라에 컨트롤바를 추가**한다. 클릭·드래그는 호스트 창 클릭 관통 덕에 실제 DOM이 그대로 받으므로(§공통-2-2) 미러는 시각 전용으로 충분하다 — 전체화면 타임코드·스크럽 오버레이가 이미 같은 방식으로 동작 중이다.
3. **(b) 스냅 인셋**: 여백이 부족해 축소가 필요한 경우, 현재 코드는 컨트롤바 CSS 트랜지션(0.3s)의 **중간 위치를 매 프레임 읽어** wrapper·mpv 네이티브 창을 계단식으로 따라 줄인다(아래 근거). 네이티브 창 리사이즈는 CSS처럼 부드러울 수 없고 매 스텝마다 OS가 직전 프레임을 새 크기로 늘려 보여줘 "찌글찌글"이 된다. 인셋을 **트랜지션과 무관한 최종값(컨트롤바 높이 기반 상수)** 으로 바꿔 **축소를 1회 스냅**으로 만든다 — 애니메이션 추적을 포기하는 대신 비율 왜곡을 제거하는 의도적 트레이드오프. (c)의 어긋남도 0.3초 내내 → 스냅 순간 1회로 실질 해소.
4. 부족분만 부분 축소하는 최적화, 동기화 IPC 원자화는 범위 외(부록 C).

### (A) `getMpvFullscreenControlsInset()` 교체 (6266~6281, 앵커: `function getMpvFullscreenControlsInset()`)
```js

    if (!document.body.classList.contains('mpv-pilot-mode')) return 0;
    if (!document.body.classList.contains('app-fullscreen')) return 0;
    if (!document.body.classList.contains('show-controls')) return 0;

    if (!controlsRect || controlsRect.height <= 0) return 0;

    // 피드백 28(b): 트랜지션 중간 위치(controlsRect.top)를 읽으면 wrapper가 0.3초 동안
    // 매 프레임 계단식으로 mpv 네이티브 창을 리사이즈해 비율이 깨져 보인다.

    // 생기므로 줌 배율을 반영해 계산한다. 중앙 고정 상태(shouldCenterVideo)에서
    // 영상은 wrapper 중앙 기준 scale 배율로 표시된다(applyVideoZoom의
```

공식 근거(검증 완료): `applyVideoZoom()`(4936~4960)이 `video.style.transformOrigin = 'center center'`를 설정하고, `shouldCenterVideo()`가 참이면 `state.videoPanX/Y = 0`으로 리셋한다(4938~4941). `getVideoRenderArea()`는 줌 무반영 순수 aspect-fit이며 그 결과는 항상 wrapper 중앙 정렬(`offsetY = (H−h)/2` → 중심 = H/2)이므로, 화면상 영상 하단 = `wrapperH/2 + (renderArea.height × scale)/2`. `scale = 1`이면 기존 `renderArea.top + renderArea.height`와 동치다. mpv 쪽도 `getMpvVideoTransform()`(6288~)이 동일 배율(`zoom = log2(scale)`)을 네이티브에 적용하므로 화면 결과가 같다.

안정성 논증(구현 후 이 성질이 유지되는지 확인할 것):
- **첫 표시 시점** wrapper는 아직 100vh(인셋 변수 0)라 `bottomGap`이 정확한 전체 여백이고, 게이트 판정은 그 뒤 재동기화에서도 같은 방향으로 유지된다 — ① 게이트 성립: wrapper 불변 → 다음 샘플도 동일 판정. ② 게이트 불성립: wrapper가 `inset`만큼 축소 → 새 `renderArea`·`scaledBottom` 재계산에서 `bottomGap`은 더 작아지거나 같음(scale ≤ 1) → 여전히 불성립, `inset` 자체는 높이 기반 상수라 동일 → CSS 변수 불변 → RO 재발화 없음. **어느 쪽도 진동하지 않는다.**
- 기존 대비 동작 등가성: 표시 완료 시점의 기존 계산값(`viewportBottom - min(controlsTop, seekbarTop)`)과 새 계산값(`height + protrusion`)은 동일하다(최종 상태에서 `controlsTop = viewportBottom - height`). 달라지는 것은 "트랜지션 중간에도 최종값을 반환한다"는 점뿐.
- `mpv-runtime-source.test.js:184~185`의 기존 정규식은 클래스 가드 3개 → `controlsBar gBCR` → `fullscreenSeekbar gBCR` 순서 존재만 요구(`[\s\S]+` 매칭)하므로 위 교체 후에도 통과한다. 교체 후 반드시 실행으로 확인.

## 엣지 케이스 & 주의사항
- **게이트 성립 케이스(여백 충분)**: 화면·영상·드로잉 완전 불변, 컨트롤바 미러만 여백 위에 나타남 → (b)(c) 원천 소멸. 실제 클릭·시크바 드래그·버튼 호버는 mpv 뒤의 진짜 DOM이 받는다(§공통-2-2). **hover 시각효과(버튼 밝아짐 등)는 style 변이가 아니어서 미러에 즉시 반영되지 않는다** — 알려진 제약으로 수용(클릭 동작 자체는 정상). 재생 중에는 진행률 바의 style 변이가 (B-2) 셀렉터에 매칭되어 미러가 프레임 주기로 갱신된다.
- **표시 중 재생 성능 비용 (의도된 트레이드오프)**: 전체화면 + 컨트롤바 표시 + 재생 중에는 진행률 변이마다 미러 전체 재동기화(드로잉 PNG 재인코딩 포함)가 돈다. 1080p에서는 프레임당 수 ms 수준으로 수용 가능하나 4K에서는 컨트롤바 표시 동안 UI가 굼떠질 수 있다(영상 재생 자체는 별도 프로세스인 mpv가 하므로 끊기지 않음). 하단 호버 동안만의 일시 비용이라 수용하되, 검증 5에서 관찰 결과가 나쁘면 부록 C-2(드로잉 PNG 캐시)를 후속으로 승격할 것 — **이번 작업에서 즉흥적으로 캐시를 구현하지 말 것**.
- **정지 상태에서의 슬라이드 인 미러**: 재생 중에는 진행률 변이 덕에 미러가 부드럽게 따라오지만, **정지 상태에서 컨트롤바가 올라올 때는 변이가 없어 미러가 동기화 시점들(즉시/rAF×2/250ms/350ms)의 중간 위치 스냅샷으로 계단식 등장**한다. 코스메틱 제약으로 수용 — 350ms 시점(C)이 최종 위치를 보장한다.
- **게이트 불성립 케이스(여백 부족: 영상이 세로를 꽉 채움, 줌>100%, 중앙고정 해제)**: 기존처럼 wrapper가 줄되 **1회 스냅**으로 바뀐다. "스르륵 작아지는" 기존 연출은 사라진다 — (b) 해소를 위한 의도적 트레이드오프(설계 결정 3). 이 경로에서 컨트롤바는 wrapper 밖 실제 DOM으로 보이고, 미러는 비겹침 스킵으로 생성되지 않는다(이중 표시 없음).
- **HTML5 엔진 불변**: 인셋 함수 첫 가드(`mpv-pilot-mode`)와 wrapper 축소 CSS(`mpv-pilot-mode` 한정)가 기존 그대로라 HTML5 전체화면(축소 없이 DOM 겹침)은 영향 없다.
- **숨김 경로 불변**: `show-controls` 제거 시 가드가 0을 반환하는 기존 동작 유지. (C)의 350ms 동기화는 숨김에도 실행되지만 컨트롤바가 opacity 0이 되므로 미러가 스킵돼 정리 효과만 있다(무해).
- **좌우 필러박스 영상(세로가 김)**: 100% 배율에서는 `bottomGap = 0`이라 게이트 불성립 → 기존 축소(하단 여백이 없으므로 올바름). 단 배율 축소 상태(zoom<100)에서는 `bottomGap = wrapperH×(1−scale)/2 > 0`이 되어 여백이 충분하면 게이트가 성립하는데, 이 역시 요청 취지("배율이 100%가 아니라 여백이 있으면")에 맞는 올바른 동작이다.
- **줌 인디케이터 등 기존 미러와의 z 순서**: 미러 컨테이너(`#htmlOverlay`) 내 순서는 배열 순서를 따른다 — 컨트롤바를 배열 끝에 추가했으므로 다른 오버레이 위에 그려진다(하단 고정이라 실제 겹침 없음).
- **전체화면이 아닐 때**: 인셋 함수 가드·컨트롤바 미러(일반 화면 컨트롤바는 wrapper 밖 하단 고정 → 비겹침 스킵) 덕에 완전 무영향. 단 일반 화면에서 wrapper와 컨트롤바가 겹치는 레이아웃이 존재하는지 확인하고(창을 아주 작게 줄인 경우 등), 겹침이 발견되면 (B-1)의 추가 요소를 `state.isFullscreen` 조건부로 감쌀 것: `...(state.isFullscreen ? [elements.controlsBar] : [])`.
- **13-24 계획서와의 충돌 없음**: 그 계획은 이 함수들을 건드리지 않는다.

# 부록 C — 범위 외 후속 권장 (이번에 구현하지 말 것 — 기록만)

1. **여백 부족 시 부분 축소**: width-bound(위아래 레터박스) 영상에서 wrapper를 `2×(inset − bottomGap)`만 줄이면(중앙 정렬이라 축소분이 위아래 반씩 분배) 영상 하단이 정확히 컨트롤바 위에 닿는다. 게이트(전부/전무)보다 정밀하지만 height-bound 분기·검증 부담이 있어 후속으로.
2. **드로잉 미러 PNG 인코딩 캐시**: `getCompositedDrawingOverlayDataUrl()`(app.js 6364~6394)이 동기화마다 영상 해상도 캔버스 합성+PNG `toDataURL`을 동기 실행한다. 스트로크 불변 시 dataURL 캐시(더티 플래그), bounds-only 변경 시 rect만 갱신하는 분리로 27의 드래그 버벅임·28 비게이트 경로의 프레임 드랍을 줄일 수 있다.
3. **오버레이 동기화 IPC 원자화**: embed bounds + overlay bounds + 미러 상태를 단일 IPC로 묶어 메인 프로세스에서 일괄 적용하면 (c)류 시차가 구조적으로 사라진다.
4. **select 도구 hover 커서**: 선택 영역 위 `move`/밖 `crosshair` 커서. 반드시 값이 바뀔 때만 `canvas.style.cursor`를 쓰는 캐싱으로 구현할 것(무캐싱 재설정은 새 깜빡임을 만든다).
5. **오버레이 호스트 `updateBounds` early-return**: bounds 불변 시 `setBounds`+`moveTop` 생략(현재 팬 드래그 중 매 프레임 실행).
6. **도구 버튼 title 단축키 힌트 정리**: index.html 도구 버튼 title의 P/L/A/R/O 힌트가 미구현 스테일 상태(L/A/O는 다른 기능에 점유됨). V 구현(작업 2)과 함께 일관화 필요.
