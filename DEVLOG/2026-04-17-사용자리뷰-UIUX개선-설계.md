## 배경 및 목적

스튜디오 팀원의 사용자 리뷰에서 수집한 UI/UX 이슈 4건 중 **코멘트 댓글 수정**은 최근 해결되어 제외. 나머지 3건은 실제 리뷰 작업 흐름(드로잉 마크업, 타임라인 스크럽, 댓글 길이 조절)을 반복적으로 방해하는 요소로, 이번 설계에서 통합 해결한다.

각 이슈는 독립적으로 수정 가능하지만 사용자 인터뷰와 비주얼 검토를 통해 확정된 단일 설계 문서로 묶어 관리한다. 구현 단계에서는 Phase 1 → 2 → 3 순서로 진행한다.

## Phase 1: 드로잉 모드 B 키 순수 토글

### 목표
B 키를 누르면 드로잉 모드가 현재 상태와 무관하게 토글되어야 한다. 현재는 OFF → ON만 가능하고 ON → OFF가 불가능한 반쪽짜리 토글이다.

### 수정 파일
- `renderer/scripts/app.js:6583-6592` (키 핸들러)
- `CLAUDE.md:117` (단축키 표 문서 정정)
- `renderer/index.html:627` (`<span id="btnDrawModeHint">D</span>` → `B` 텍스트 정정. UI에 표시되는 단축키 라벨)

### 구현 내용

**현재 코드:**
```javascript
if (userSettings.matchShortcut('drawMode', e)) {
  e.preventDefault();
  if (!state.isDrawMode) {        // ← 버그: OFF일 때만 토글
    toggleDrawMode();
  }
  const brushBtn = document.querySelector('.tool-btn[data-tool="brush"]');
  if (brushBtn) brushBtn.click();
  return;
}
```

**변경안:**
```javascript
if (userSettings.matchShortcut('drawMode', e)) {
  e.preventDefault();
  const wasOff = !state.isDrawMode;
  toggleDrawMode();                // 항상 토글
  if (wasOff) {
    const brushBtn = document.querySelector('.tool-btn[data-tool="brush"]');
    if (brushBtn) brushBtn.click();  // 진입 시에만 브러시 자동 선택
  }
  return;
}
```

**CLAUDE.md 수정:**
```diff
- | `D` | 그리기 모드 |
+ | `B` | 그리기 모드 |
```

### 동작 명세 (순수 토글)
| 현재 상태 | B 누름 | 결과 |
|---|---|---|
| 드로잉 OFF | B | ON + 브러시 선택 |
| 드로잉 ON (브러시 사용 중) | B | OFF |
| 드로잉 ON (지우개 등 다른 도구) | B | OFF (도구 상태 보존) |

### 예상 리스크
- **낮음**. 단순 조건 제거로 기존 동작은 그대로 유지되고 누락된 OFF 경로만 추가.
- 다른 단축키와 충돌 없음 (B는 현재 drawMode에만 할당).

## Phase 2: 타임라인 프레임 격자 단계식 표시 + 격자 토글

### 목표
1. 줌인 중 "격자가 없다가 갑자기 꽉 참"이라는 불연속 전환을 **계단식 밀도 증가**로 해소
2. 최소 줌(pxPerFrame ≈ 0.2)에서 1초 강조선이 너무 촘촘한 문제 해소
3. 사용자가 격자 전체를 끌 수 있는 **master 토글** 제공 (기본 ON)

### 수정 파일
- `renderer/scripts/modules/timeline.js:776-817` (`_updateFrameGrid`), `824-861` (`_renderFrameGrid`)
- `renderer/styles/main.css` (½초/¼초 격자선 색)
- 툴바 HTML/JS (격자 표시 토글 버튼 추가)
- `renderer/scripts/modules/user-settings.js` (토글 상태 영속화)

### 구현 내용

**기존 → 신규 전환**: 현재 `_updateFrameGrid`의 3단 분기(2px 미만 숨김 / 2~4px 5프레임 / 4px+ 개별)를 **제거하고 신규 5단 계단식 로직으로 완전 교체**한다. 기존 `every5` 레이어는 삭제.

**단계 임계값 (하드 스냅, 페이드 없음):**

| pxPerFrame | 표시 단계 |
|---|---|
| < 0.5 | 5초 강조선만 (1초 선 숨김) |
| 0.5 ≤ pxPerFrame < 1.0 | + 1초 |
| 1.0 ≤ pxPerFrame < 2.0 | + ½초 |
| 2.0 ≤ pxPerFrame < 4.0 | + ¼초 (6프레임 @24fps, fps 인식으로 일반화) |
| pxPerFrame ≥ 4.0 | + 매 프레임 |

**fps 일반화**: ¼초 선은 `Math.round(fps / 4)`프레임마다. 24fps → 6f, 30fps → 7~8f. ½초는 `Math.round(fps / 2)`.

**색상**:
- 5초/1초 강조선: 기존 `rgba(255,220,100,0.75)` 유지
- ½초: `rgba(120,180,255,0.55)` (신규, 연파랑)
- ¼초: `rgba(120,180,255,0.32)` (신규, 더 연한 파랑)
- 매 프레임: 기존 `rgba(255,255,255,0.22)` 유지 (하드 스냅)

**렌더링 방식**: 기존처럼 CSS `background` 다층 `linear-gradient`로 구현 (성능 유지). 각 단계를 독립 레이어로 합성.

**Master 토글**:
- DOM 위치: `renderer/index.html:716`의 `.timeline-toolbar-right` 내부에 아이콘 버튼(id=`btnGridToggle`) 신규 삽입. 격자 아이콘 + 활성 상태 하이라이트.
- 상태 필드: `state.gridVisible` (기본 `true`)
- 영속화 키: `userSettings.timeline.gridVisible` (`user-settings.js`의 `timeline` 섹션에 추가)
- OFF 시 `frameGridContainer.style.display = 'none'`. 렌더 자체 스킵.

**히스테리시스 (MVP 포함)**: 단계 경계 근처에서 줌 중 깜빡임 방지. 각 임계값에 ±3% 버퍼 도입:
```
임계 = 0.5 → 진입 0.515, 이탈 0.485
```
이전 단계 상태를 저장해(`state._lastFrameGridTier`) 재진입 때만 전환. 깜빡임 회귀 위험이 높아 MVP에 포함.

### 예상 리스크
- **중간**: `_renderFrameGrid`는 CSS 배경 기반이라 다층 `linear-gradient` 문자열이 길어짐. 캐시 무효화 조건(zoom 변경) 유지 필요.
- **낮음**: 토글 OFF 상태가 기본값(ON)과 달라 팀원 혼란 가능 → 툴바에서 명확히 보이도록 시각 힌트.
- fps가 24/30이 아닌 경우(예: 25fps) 분할 수가 어색해질 수 있음 → `Math.round`로 가장 가까운 정수 프레임 사용.

## Phase 3: 코멘트 중첩 하이브리드 처리 (Mode C)

### 목표
여러 댓글이 시간 구간상 겹칠 때, 뒤쪽 댓글의 좌·우 리사이즈 핸들에 접근할 수 없는 문제를 해소. 평소 UI는 현재처럼 컴팩트하게 유지하되, 필요 시 해당 구간만 레인으로 펼쳐 편집 가능하게.

### 수정 파일
- `renderer/scripts/modules/timeline.js:1951-2003` (구간 댓글 렌더), `1553-1713` (클러스터링 로직 확장)
- `renderer/styles/main.css:2865-2971` (코멘트 트랙 높이 애니메이션, 클러스터 배지 스타일)
- `renderer/scripts/app.js:3155-3233` (핸들 이벤트 — 변경 없음, 접근성만 확보)

### 구현 내용

**기존 핸들/리사이즈 로직은 재사용**. 병목은 단일 row 중첩이며 데이터 모델(`startFrame`/`endFrame`/`setDuration()`)과 핸들 DOM(`.comment-handle-left/right`)은 이미 존재함.

**클러스터 체계 분리 (중요)**: 기존 점 댓글용 픽셀 거리 클러스터링(`timeline.js:1553-1622`, `minClusterDistance=20px`)은 **그대로 유지**하고, 이번에 추가되는 것은 **구간 댓글(`.comment-range-item`) 전용 프레임 교차 클러스터링**. 두 체계는 대상이 달라(점 vs 구간) 충돌 없음. 점+구간 혼합 처리는 본 설계 범위 밖 — 혼합된 경우 점 댓글은 기존 로직, 구간 댓글은 신규 클러스터로 각각 분리 처리.

**클러스터 식별자(`clusterId`)**: 클러스터에는 안정적 id가 없으므로 `cluster[0].markerId`(시작 프레임이 가장 이른 댓글의 id)를 클러스터 키로 사용. 재렌더 시 동일 클러스터 인식 기준. 해당 댓글이 삭제/수정되면 펼침 상태 자동 해제.

**A. 클러스터 탐지 (연결 성분)**
```javascript
// 시간 구간이 교차하는 댓글을 그룹으로 묶음
function findClusters(comments) {
  const sorted = [...comments].sort((a, b) => a.startFrame - b.startFrame);
  const clusters = [];
  let current = [], currentEnd = -1;
  sorted.forEach(c => {
    if (current.length === 0 || c.startFrame < currentEnd) {
      current.push(c);
      currentEnd = Math.max(currentEnd, c.endFrame);
    } else {
      clusters.push(current);
      current = [c];
      currentEnd = c.endFrame;
    }
  });
  if (current.length) clusters.push(current);
  return clusters;
}
```

**B. 접힌 상태 (기본 렌더링)**
- 클러스터 크기 1 → 기존처럼 단일 바/점으로 렌더
- 클러스터 크기 ≥ 2 → 주황 배지 바 렌더:
  - 배경: `linear-gradient(180deg, rgba(245,158,11,0.9), rgba(217,119,6,0.85))` (`--warning` 계열)
  - 좌측 3색 "스택 힌트" (내부 댓글 3개 작성자 색 샘플)
  - 가운데 `⊕N` 칩 (클러스터 크기)
  - 우측 "클릭 → 펼치기" 안내 텍스트
- 접힘 상태에서도 배지 호버 시 내부 댓글 요약 말풍선 표시 (기존 `.comment-marker-tooltip` 재사용)

**C. 펼침 애니메이션**
- 배지 클릭 → `state.expandedClusterId = cluster.id`
- 해당 클러스터 범위에 대해 그리디 레인 할당:
  ```
  lanes = []
  for each comment in cluster (sorted by startFrame):
    find lowest lane where no overlap with existing comments in that lane
    push to lane
  ```
- `comment-track`의 `height`를 `34px → max(34, maxLane × 30 + 6)px`로 CSS `transition: height 0.28s ease`
- 각 댓글 바를 해당 레인의 top 좌표에 배치

**D. 접기**
- 펼쳐진 구간 하단에 "▲ 접기" 칩 표시 (`.collapse-btn` 클래스)
- 칩 클릭 또는 타임라인 빈 영역 클릭 시 `expandedClusterId = null`
- **MVP 제외**: 재생 헤드가 클러스터 범위를 벗어날 때 자동 접힘은 본 설계의 MVP에서 제외. 수동 접기(칩/외부 클릭)만 지원. 추후 사용성 관찰 후 별도 이슈로 검토.

**E. 접근성/상호작용**
- 펼쳐진 각 바에서 기존 핸들 이벤트(`app.js:3155-3233`)가 그대로 동작 — 수정 불필요
- 키보드 탐색: 배지에 `tabindex="0"`, Enter로 펼침/접힘
- 스크린리더 라벨: `aria-label="3개의 겹친 코멘트. 펼치려면 Enter"`

**F. 기존 동작 유지**
- 비중첩 점 코멘트(`.comment-marker-track`) 모양 변경 없음
- 비중첩 구간 바(`.comment-range-item`) 모양 변경 없음
- 구간 바 리사이즈 로직 변경 없음 — 펼쳐진 상태에서도 동일

### 예상 리스크
- **중간**: 클러스터 배지가 기존 단일 클러스터 마커(점 댓글 주황)와 시각적으로 가까움 → 스택 힌트와 `⊕N` 칩으로 구별
- **중간**: 펼침 상태에서 재생 중 재생 헤드가 클러스터 밖으로 이동 시 자동 접힘 타이밍 결정. 너무 빠르면 편집 방해, 너무 늦으면 레이아웃 고정 느낌.
- **낮음**: 클러스터링 계산 비용 — 댓글이 수백 개 단위에서 매 렌더마다 O(n log n). 기존 마커 클러스터링도 비슷한 수준이라 큰 문제 아님.
- **낮음**: 펼침 상태에서 Liveblocks 실시간 동기화로 다른 사용자가 해당 클러스터의 댓글을 수정/삭제 시 → 펼침 상태에서 부드럽게 재계산 (React처럼 key 기반 재사용).

## 테스트 방법

### Phase 1 (수동)
1. 드로잉 OFF 상태 → B 누름 → 드로잉 모드 진입 + 브러시 선택 확인
2. 드로잉 ON (브러시) → B 누름 → OFF 확인
3. 드로잉 ON → 지우개 선택 → B 누름 → OFF 확인 (도구 상태 무관)
4. 오디오 모드에서 B 누름 → 무반응 확인 (기존 차단 로직 유지)

### Phase 2 (수동 + 비주얼)
1. 휠 줌 0.2 → 8.0 px/frame 스윕:
   - 0.2 근처: 5초 강조선만 보이고 깔끔
   - 0.7 근처: 1초 선 등장
   - 1.5 근처: ½초 선 추가
   - 2.5 근처: ¼초 선 추가
   - 5.0 근처: 매 프레임 선 추가
   - 각 전환이 깜빡임 없이 이뤄지는지
2. 격자 표시 토글 OFF → 격자 완전 숨김 확인
3. 토글 상태가 앱 재시작 후에도 유지되는지 (`userSettings` 영속화)
4. 25fps, 30fps 비디오에서 ½초/¼초가 적절한 프레임 간격으로 표시되는지

### Phase 3 (수동 + 협업 테스트)
1. 동일 프레임 근처에 구간 댓글 3개 생성 → 주황 배지 + `⊕3` + 스택 힌트 확인
2. 배지 클릭 → 부드러운 애니메이션으로 레인 3단 펼침 확인
3. 펼쳐진 각 바 호버 → 좌우 핸들 나타남, 드래그로 길이 조절 성공
4. "▲ 접기" 클릭 → 접힘 복귀
5. Liveblocks 협업 상태에서 한 사용자가 펼친 클러스터를 다른 사용자가 수정 시 레이아웃 깨짐 없음
6. 비중첩 댓글들은 모양 변경 없이 그대로 동작하는지 (회귀 확인)
